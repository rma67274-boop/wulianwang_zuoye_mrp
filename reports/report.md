## 1. 架构对比分析（重点）

### 1.1 传统 RAG vs Karpathy 方案

| 维度 | 传统 RAG | Karpathy 长上下文方案（本项目） |
| --- | --- | --- |
| 数据流程 | 分块 -> 向量化 -> 检索Top-K -> 生成 | raw -> LLM整库阅读 -> 结构化Markdown |
| 召回特性 | 依赖检索，存在漏召回风险 | 整库进入上下文，召回更完整 |
| 跨文档推理 | 受Top-K片段限制 | 能在单次上下文中跨页面综合 |
| 系统复杂度 | 需要向量库和检索链路 | 文件系统 + LLM API 即可 |
| 可维护性 | 索引与文档双维护 | 只维护 Markdown |
| 适用规模 | 中大规模知识库 | 个人/课程级中小规模 |

### 1.2 设计决策说明

1. 采用 Markdown 作为唯一知识载体。
原因：符合题目要求、可在 Obsidian 直接可视化、便于人工审查和版本管理。

2. 不引入向量数据库。
原因：课程作业规模较小，优先保证架构简单与可解释，减少非必要工程复杂度。

3. 提供两条编译路径。
- 基础路径：`src/build_kb.py`，快速生成稳定样例库。
- 自动路径：`src/compile_raw_llm.py`，满足“LLM 自动读取 raw 后编译”的作业要求。

4. 问答阶段使用整库上下文。
`src/qa.py` 会加载 `wiki/` 全部页面并发送给模型，保持“非检索、长上下文”原则。

### 1.3 长上下文模型局限性讨论

1. Token 成本高。
整库输入会显著增加输入 token，编译和问答都可能产生较高调用费用。

2. 响应延迟增加。
上下文变长后，模型推理时间增加，交互体验不如短提示。

3. 上下文窗口上限约束。
当知识库持续增长，最终会超过单次窗口，需要分层编译或分主题索引。

4. 输出稳定性受 Prompt 影响。
若未限制输出格式，模型可能出现页面结构不一致、链接缺失等问题。

## 2. Prompt 工程细节

### 2.1 Compilation Prompt（最终版核心要点）

在 `src/compile_raw_llm.py` 中，编译 Prompt 的核心要求为：

1. 只输出 JSON，不输出解释性文字。
2. 结构固定为：`site_title + pages[]`。
3. 页面字段包含：`title/summary/highlights/equations/questions/tags/related`。
4. `pages` 数量至少 10。
5. `related` 必须引用已存在页面标题。
6. 语言为中文，主题聚焦电磁场与电磁波课程。

### 2.2 Prompt 迭代过程

1. 第一版问题：输出不稳定。
现象：模型容易输出带解释文字的自然语言，JSON 解析失败。
改进：加入“只输出合法 JSON”并启用 `response_format=json_object`。

2. 第二版问题：页面数量不足。
现象：有时只生成 6 到 8 页。
改进：在 Prompt 明确写“pages 数量 >= 10”。

3. 第三版问题：关联链接失效。
现象：`related` 包含不存在页面标题。
改进：增加后处理校验（过滤不存在标题），并在 Prompt 中强调关联约束。

4. 第四版问题：内容跑题。
现象：个别页面偏离课程主题。
改进：补充“主题聚焦电磁场与电磁波课程”的硬约束。

## 4. 性能评估

### 4.1 知识库规模

实测（当前项目）：

- Markdown 文件数：11（索引 1 + 章节 10）
- 总字符数：6297
- 中文字符数：3018
- 英文/数字词数：336
- token 估算：约 3704（按 `chars/1.7` 粗略估算）

### 4.2 编译耗时与问答响应时间

本地实测（PowerShell Measure-Command）：

- `python src/build_kb.py`：约 0.095 s
- `python src/qa.py "高斯定律和静电场有什么关系"`：约 0.222 s（本地兜底模式）

说明：

1. 上述耗时是“本地文件处理/本地兜底”性能。
2. 使用远程长上下文模型时，编译与问答耗时会显著上升，取决于模型和网络。

### 4.3 API 调用成本估算

采用统一估算公式：

`单次成本 = 输入token/1e6 * 输入单价 + 输出token/1e6 * 输出单价`

示例估算（仅示例，实际以平台计费为准）：

1. 假设单次问答输入 4k token、输出 800 token。
2. 假设输入单价 2 元/百万 token，输出单价 8 元/百万 token。

则：

- 输入成本：`4000/1e6 * 2 = 0.008` 元
- 输出成本：`800/1e6 * 8 = 0.0064` 元
- 单次总成本：约 `0.0144` 元

若课程演示进行 30 次问答，约 `0.43` 元。

## 5. 反思与展望

### 5.1 适用与不适用场景

适合：

1. 个人学习型知识库（课程笔记、项目文档）。
2. 文档规模在长上下文窗口可承受范围内。
3. 强调可读、可维护、可在 Obsidian 中可视化的场景。

不适合：

1. 百万字以上且持续高速增长的文档库。
2. 严格低延迟在线问答场景。
3. 对成本和吞吐敏感的高并发系统。

### 5.2 若知识库增长到 100 万字的调整策略

1. 分层编译：先按主题子库编译，再维护总索引页。
2. 轻量索引：不引入向量库，可先做关键词索引作为路由层。
3. 增量更新：仅重编译受影响页面，减少全量 token 消耗。
4. 缓存机制：缓存高频问题答案与上下文摘要。
5. 成本控制：对编译任务设置批处理窗口，避免频繁全量调用。

## 6. 结论

本项目已完成 Karpathy 风格长上下文知识库的主要功能闭环：

1. 结构化 wiki 页面（Markdown 唯一存储）
2. 长上下文问答（非向量检索）
3. 图谱导出与 Obsidian 兼容
4. 自动编译入口（LLM 读取 raw）

对课程作业目标而言，本方案在“架构简洁性、可解释性、可演示性”方面表现较好，后续可重点加强增量编译与真实 API 评测。
